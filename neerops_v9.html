00 Goal-First Architecture — Design Constraints That Cannot Be Violated
Architectural filter: Every component, layer, and feedback loop in this document is justified by at least one of G1 / G2 / G3 / LLMOps. Components that do not serve one of these four goals are not in this architecture — regardless of how technically interesting they are.
G1 · Smooth Deployment
Every PR travels from webhook to production with zero manual steps. The pipeline is a statistical decision engine, not a timed sequence. A bad deploy is caught at 5% canary traffic before it reaches users at scale. Build failures, security blocks, and infrastructure timeouts all have explicit recovery paths — the pipeline never hangs silently.
Layers: L1 Understanding · L2 Review · L3 Build · L4 Deploy
Guards: Bayesian canary · Dead-man's switch · Orchestrator FSM
Security: Gitleaks · Trivy · ZAP · Cosign signing
G2 · 24/7 Uptime
The system must remain operational regardless of deployment activity, infrastructure changes, secret rotation, or node maintenance. Monitoring is continuous and parallel to deployments — not sequential. Anomaly detection runs three tiers (static threshold → 3σ → Bayesian ML). If the monitor itself fails, a circuit breaker activates threshold-only fallback rules immediately.
Layers: L5 Monitor · L6 Healing · Closed-loop feedback
Guards: Circular heal detection · Auto-rollback SLA · Vault rotation lock
Failsafe: T+10 safe rollback · T+15 unconditional rollback
G3 · Auto-Heal + LLMOps
When anything breaks, the system heals itself before a human notices. Heuristics resolve 90% of events in ~5 seconds. RL handles variations of known patterns. LLMs diagnose genuinely novel failures and convert each resolution into a new heuristic rule — teaching themselves out of a job. L9 continuously improves the prompts, model choices, and alert thresholds autonomously.
Layers: L6 4-tier healing · L9 Meta-Learning · L0 Cognition
LLMOps: Prompt Evolver · Model Selector · Token budget mgmt
Feedback: Real-time heuristic update · KL-drift detector

02 Centralized Orchestrator — The Only State Owner
The Orchestrator owns the PR state machine. No layer can change system state directly — they publish results to the Event Bus and the Orchestrator decides what happens next. This is not a naming convention: layers literally cannot see each other's internal state.
Rule: Every layer publishes a LayerResult event to the Event Bus. The Orchestrator consumes it, updates the state machine, and emits the next LayerCommand. A layer that calls another layer directly is a bug — not a design choice.
Orchestrator Responsibility Implementation What Breaks Without It
Orchestrator High Availability Active-passive Orchestrator pair. Leader election via Redis Redlock (quorum of 3 Redis nodes). Standby takes over in <5 sec on leader heartbeat miss. All FSM state stored in Redis + PostgreSQL — zero in-memory state. On failover, standby replays the last 10 events from Event Bus consumer group to reconstruct mid-flight PR states. Split-brain guard: only the elected leader may publish LayerCommand events (enforced by token in Redis). Orchestrator becomes SPOF: one process crash pauses all active deployments with no recovery path. Split-brain causes duplicate deploys.
Redis High Availability Redis Sentinel with 3 sentinels + 1 replica per shard, or Redis Cluster for horizontal scale. FSM state uses append-only Redis Streams (durable). On partition: Orchestrator detects sentinel failover event and pauses state transitions until quorum re-established — no split-brain write. CRDT-compatible lock keys use SET NX with TTL; on partition recovery, expired locks are automatically released, not double-acquired. Redis split-brain corrupts canary lock state. Two Orchestrator instances believe they own the same PR. Duplicate canary deploys contaminate metrics.
Event Bus High Availability Redis Streams with AOF persistence (appendfsync everysec). Stream consumer groups guarantee at-least-once delivery — a message is not acknowledged until the consuming layer emits a LayerResult. On Redis failover, pending messages in the PEL (Pending Entry List) are claimed by standby consumer after 30s timeout. Dead-letter stream captures messages that fail 3 delivery attempts — Orchestrator alerts and routes to human review. Event Bus loss mid-pipeline orphans PRs in intermediate states with no recovery path.
PR State Machine Redis-backed FSM. States: DETECTED → UNDERSTANDING → GOAL_CHECK → REVIEWING → BUILDING → STAGING → CANARY_5 → CANARY_50 → ROLLING_100 → MERGED | FAILED | ROLLED_BACK Without a single FSM, two layers can both believe they are "in charge" of the same PR → duplicate deploys, contradictory rollbacks.
Per-service namespace lock Redis SETNX lock on lock:{service}:{namespace}. Held for canary duration. Second PR for same service queues behind first. Two concurrent PRs for the same service contaminate each other's canary metrics. Impossible to attribute an anomaly to the correct PR.
Dead-man's switch Every kubectl / helm operation has a 5-min timeout watchdog. On timeout: publish OperationTimeout → Orchestrator triggers immediate rollback. Kubernetes API unavailability mid-canary would hang the pipeline forever with no recovery.
Retry budget Per-PR, per-layer retry counter in Redis. Max 3 retries with exponential backoff (2s, 8s, 32s). On exhaust: emit LayerFailed, skip to Healing. Unlimited retries on a build failure waste resources and block all other PRs queued behind.
Circuit breakers Per-layer circuit breaker: open on 3 failures in 5 min. Half-open: probe after 60s. If Layer N circuit is open, Orchestrator uses heuristic fallback and alerts. A broken Build layer would be retried continuously, burning compute and blocking all other PRs.
Human SLA timer On CRITICAL escalation: Orchestrator sets a 15-min countdown. At T+10min: attempt safe rollback. At T+15min: unconditional rollback + QLDB SLA breach record. Human does not respond → system stays degraded indefinitely with no recovery path.
Vault rotation lock Orchestrator defers Vault secret rotation during CANARY_* state using two concrete mechanisms: (1) vault write sys/leases/renew/{lease_id} increment=3600 — extends the existing lease TTL by 1 hour, preventing the scheduled rotation from firing; (2) A custom Vault policy sentinel attached to the NEEROps role blocks rotation writes when the Orchestrator sets a neerops/canary-lock/{service} KV flag. Lock is released on MERGED or ROLLED_BACK via vault kv delete neerops/canary-lock/{service}. If the lock is not released within 2 hours (circuit break on abandoned canary), a Vault watchdog alert fires for manual resolution. QLDB records every lock set and release with timestamp. Vault rotates DB credentials during canary → all canary pods lose DB connection → false rollback triggered (I03).

# Bayesian canary gate — runs every 30 seconds during canary phase
def evaluate_gate(baseline_metrics, canary_metrics, prior_alpha=1, prior_beta=1):
    # Beta-Binomial model on error counts
    canary_successes   = canary_metrics.requests - canary_metrics.errors
    baseline_successes = baseline_metrics.requests - baseline_metrics.errors

    posterior_canary   = Beta(prior_alpha + canary_successes,   prior_beta + canary_metrics.errors)
    posterior_baseline = Beta(prior_alpha + baseline_successes, prior_beta + baseline_metrics.errors)

    # Monte Carlo comparison (10,000 samples)
    samples_canary   = posterior_canary.sample(10000)
    samples_baseline = posterior_baseline.sample(10000)

    p_canary_better = (samples_canary > samples_baseline).mean()          # P(canary ≥ baseline)
    p_canary_worse  = (samples_canary < samples_baseline * 0.99).mean()  # P(canary clearly worse)

    # L8 World Model adjusts these thresholds based on risk score
    if p_canary_better >= 0.95:  return "ADVANCE"   # conclusive: new ≥ baseline
    if p_canary_worse  >  0.90:  return "ROLLBACK"  # conclusive: new is worse
    if elapsed_minutes >= 45:    return "ESCALATE"  # hard ceiling: not conclusive after 45min
    return "HOLD"                                     # not enough data yet — wait

def llm_healing_tier(anomaly, llm_client, heuristic_lib, rl_model):
    # 1. Check if heuristic or RL can handle it first
    heuristic = heuristic_lib.match(anomaly)
    if heuristic.score >= 0.5: return heuristic_lib.execute(heuristic)

    rl_action = rl_model.predict(anomaly)
    if rl_action.confidence >= 0.7: return rl_model.execute(rl_action)

    # 2. LLM tier: rate limit + budget pre-check
    if llm_client.remaining_tokens() < CRITICAL_RESERVE and anomaly.priority != 'CRITICAL':
        return escalate_to_human(anomaly, reason='llm_budget_exhausted')

    # 3. Build context: logs + metrics + current state
    plan = llm_client.diagnose(anomaly.logs, anomaly.metrics, anomaly.k8s_state)
    if plan.confidence < 0.85: return escalate_to_human(anomaly, reason='llm_low_confidence')

    # 4. Dry-run validation before any execution
    dry_run_result = k8s_client.dry_run(plan.actions)
    if not dry_run_result.ok: return escalate_to_human(anomaly, reason='dry_run_failed')

    # 5. Execute and watch for resolution
    outcome = execute_and_verify(plan.actions, timeout=600)

    # 6. On success: immediately create heuristic so LLM never handles this again
    if outcome.resolved:
        event_bus.publish('NewHeuristicRule', rule=extract_rule(anomaly, plan))

    return outcome

def lookup_heuristic(anomaly: AnomalyEvent) -> Optional[HeuristicRule]:
    # 1. Exact signature match (fastest — O(1) with Redis hash)
    exact = redis.hget("heuristics:exact", anomaly.signature_hash)
    if exact and exact.confidence > 0.5: return exact

    # 2. Semantic embedding match (pgvector cosine similarity)
    query_vec = embed(anomaly.context_text)
    result = db.execute("""
        SELECT * FROM heuristics
        WHERE 1 - (embedding <=> %s) > 0.92
          AND last_validated_at > NOW() - INTERVAL '30 days'
        ORDER BY confidence DESC LIMIT 1
    """, [query_vec])
    return result.first()  # None if no match → escalate to RL tier

def write_heuristic(anomaly: AnomalyEvent, action: HealingAction, source: RuleSource):
    rule = HeuristicRule(
        rule_id=uuid4(), anomaly_signature=anomaly.to_pattern(),
        action=action, confidence=0.7, source=source,
        last_validated_at=datetime.utcnow(),
        embedding=embed(anomaly.context_text)
    )
    db.insert("heuristics", rule)
    redis.hset("heuristics:exact", anomaly.signature_hash, rule)  # fast path cache
    ctx.logger.write("NewHeuristicRule", rule_id=rule.rule_id, source=source)  # QLDB

def build_knowledge_graph(pr_diff: PRDiff, prev_kg: KnowledgeGraph) -> KnowledgeGraph:
    changed_files = pr_diff.changed_file_paths()

    # Determine if incremental or full rebuild is warranted
    delta_ratio = len(changed_files) / max(prev_kg.node_count, 1)
    if delta_ratio > 0.20 or pr_diff.iac_drift_detected or prev_kg.is_stale:
        return build_full_kg(pr_diff)                     # full rebuild path (unchanged from v8.1)

    # Incremental path: recompute only affected subgraph
    affected_nodes = prev_kg.subgraph_reachable_from(changed_files)  # BFS on import edges
    unchanged_nodes = prev_kg.nodes - affected_nodes

    new_kg = KnowledgeGraph()
    new_kg.promote_nodes(unchanged_nodes, stale_flag=False)   # reuse unchanged nodes
    new_kg.recompute_nodes(affected_nodes, pr_diff)              # re-scan only delta

13 Hard Safety Constraints — Non-Negotiable, Non-Bypassable
These constraints cannot be overridden by any layer, including L8 and L9. They are checked by the Orchestrator before any state transition is executed. They exist because G2 (24/7 uptime) and G3 (auto-heal) require a system that is safe to run autonomously at Level 4.
S1
No deployment to production without a signed container image. Kubernetes admission webhook (Connaisseur) rejects any pod spec with an unsigned image. This gate is enforced at the k8s API level — no NEEROps layer can bypass it programmatically.
S2
No secret ever baked into an image or environment variable. Vault agent sidecar is the only path. Build layer has a post-build check: scan image layers for common secret patterns. Fail build if any found. Applies to all layers including L9.
S3
Security-critical prompt sections are SHA-256 hash-locked. L9 Prompt Evolver cannot modify any prompt section that touches: Gitleaks gate, Trivy gate, ZAP gate, secrets injection. Any attempted change to a locked section is blocked and queued for human sign-off + hash recomputation.
S4
All architecture changes proposed by L9 require human review. L9 can propose; it cannot apply. Architecture proposals are published to a human-review queue. No action is taken until explicit human approval is recorded in QLDB with a named approver.
S5
Rollback is always possible before human escalation completes. The Orchestrator maintains a "last known good" image tag for every service at all times. At T+10min of a CRITICAL escalation with no human ACK, a safe rollback to last-known-good is attempted before the hard T+15min unconditional rollback fires.
S6
Vault secret rotation is locked during active canary. Orchestrator sets a deployment lock signal to Vault when a PR enters CANARY_* state. Vault defers all scheduled rotations for that service's secrets. Lock releases on MERGED or ROLLED_BACK — never left dangling (I03 mitigation).
S7
Circular healing loop detection is mandatory. L6 Healing tracks the last 3 actions taken for each anomaly instance via action signature hash. If the same action is proposed a 3rd time, the tier escalates immediately — no third execution. This prevents infinite scale-up → OOM → scale-up cycles (I04 mitigation).
S8
QLDB audit log is append-only and cannot be modified by any NEEROps layer. The QLDB writer is a global singleton with INSERT-only capability. No layer has UPDATE or DELETE access to QLDB records. S3 Object Lock mirror is WORM — no API can delete or overwrite it for the 10-year retention period.

16 RL Training Pipeline — Off-Policy, Reward Shaping, Safe Exploration v8.1
The Goal Engine (L8) and L6 Healing RL tier use a PPO policy network trained on deployment outcomes. This section specifies the training pipeline in detail — including the off-policy/on-policy choice, reward shaping, and safe exploration constraints that were absent from v8.0.
Design choice — off-policy training: NEEROps uses off-policy training (Replay Buffer + PPO with importance sampling corrections) rather than on-policy training. Reason: on-policy training would require the RL agent to generate its own deployment decisions to collect experience, which is unsafe in production. Off-policy training learns from the historical log of heuristic, LLM, and human decisions stored in QLDB — the RL model improves from observed outcomes without having to act autonomously during training. Safe exploration is enforced by constraining the RL policy to a heuristic-bounded action space during the initial 90-day warm-up period.
Training Data
Source: QLDB deployment outcomes, healing traces, canary metric windows — 7-day rolling window. Each training example: (state = metrics+risk+context, action = deploy/hold/rollback/escalate, reward = observed outcome). Off-policy corrections via importance sampling weights (IS-weighted PPO). Minimum: 500 examples per daily training run. Fewer than 500 → skip training cycle, use previous model.
Reward Shaping
3-dimensional reward aligned with H1/H2/H3:
• H1 (deploy safely): +1.0 for clean deploy to 100%, −2.0 for rollback due to bug, −0.5 for false rollback (rollback caused by infra event, not bug)
• H2 (uptime): +0.5 for every 5-min window with error_rate < 0.1%; −1.0 for any SLA breach during or after deploy
• H3 (cost): −0.01 per LLM token spent; +0.1 for every heuristic resolution that avoids LLM call
Discounted sum γ=0.95 over a 2-hour horizon.
Safe Exploration
During 90-day warm-up: RL policy is constrained to agree with the heuristic action ≥80% of the time (KL penalty in loss function). After warm-up: constraint relaxed to 60%. Hard constraints baked into action space: RL cannot propose actions that would bypass S1–S8 safety constraints — these are structurally impossible actions, not just penalized ones. KL-divergence monitor: if RL policy distribution shifts >0.1 from previous checkpoint → auto-revert to checkpoint + L9 retraining scheduled.
Validation & Shadow Deployment
Before any new RL model enters production: 24-hour shadow deployment. Shadow model observes every decision but does not act — its proposed actions are logged to QLDB alongside the live model's actions. Promotion criterion: shadow model's proposed actions agree with human expert labels ≥90% on a held-out 200-example eval set AND have lower mean regret than the live model on shadow observations. KL-divergence between shadow and live policy: must be <0.05 to prevent destabilizing jumps.
Continuous Monitoring
Post-deployment: RL reward trend monitored by L7 Feedback every hour. If 3-hour rolling mean reward drops >15% below 7-day mean → auto-revert to previous model checkpoint + alert. Model version, reward history, and all eval results written to QLDB — full audit trail of every model version ever deployed. L9 Meta-Cognition audits RL decision consistency: if RL frequently disagrees with heuristics on cases where heuristics historically succeed → flag for retraining.
Technology Stack
PyTorch PPO implementation (Stable-Baselines3). Feature encoding: 48-dimensional state vector (16 metric features × 3 time horizons). Action space: discrete 4-action {ADVANCE, HOLD, ROLLBACK, ESCALATE}. Training: single A100 GPU, ~12 min/day. Model serialization: TorchScript for inference — no Python dependency at inference time. Inference latency: <50ms per decision. Model artifacts stored in S3 with version tags; rollback to any previous version in <30 sec.
# RL training loop — off-policy PPO with importance sampling
def train_rl_model(qldb_window: 7):
    experiences = qldb.query_deployment_outcomes(days=qldb_window)
    if len(experiences) < 500:
        return skip_cycle("insufficient_data")

    buffer = ReplayBuffer(experiences)
    model  = PPO.load_checkpoint()  # off-policy: load latest, not fresh init

    for batch in buffer.sample(batch_size=256):
        is_weights = compute_importance_sampling_weights(batch, model.policy)
        reward     = shape_reward(batch, H1_weight=1.0, H2_weight=0.5, H3_weight=0.1)
        loss       = ppo_loss(batch, is_weights, reward)

        # Safe exploration: KL penalty if diverging from heuristic during warm-up
        if warmup_period:
            loss += kl_penalty_weight * kl_divergence(model.policy, heuristic_policy)

        loss.backward(); optimizer.step()

    shadow_model = ShadowDeployment(model)
    if shadow_model.passes_24h_eval(eval_set, threshold=0.90):
        promote_to_production(model); qldb.record_model_version(model)
17 Autonomy Supervisor — Monitoring the Monitor v8.1
A system that operates at Level 4 autonomy must be able to monitor its own control plane. The Autonomy Supervisor is a lightweight, independent process that watches the health of every critical NEEROps subsystem — Orchestrator, L5 Monitor, Redis, Event Bus, RL models, and prompt quality — and acts without depending on any of the systems it monitors.
Why a separate supervisor: The Orchestrator cannot reliably report its own failure. L5 cannot alert on its own downtime. The supervisor is intentionally simple (no Redis dependency, no Vault dependency, no Event Bus dependency) — it runs as an independent k8s Deployment with its own PagerDuty direct-write credentials. Its only job is: observe heartbeats, fire alerts, and trigger safe rollback if critical systems are unresponsive.
Orchestrator Health Watch
Checks Orchestrator leader heartbeat every 5 seconds via a dedicated /health HTTP endpoint (not via Redis — independent path). On miss: (1) Triggers standby Orchestrator leader election. (2) If no Orchestrator leader is elected within 30 seconds: pauses all active canaries via direct kubectl annotation. (3) Fires PagerDuty CRITICAL. (4) Logs health event to S3 directly (not QLDB — QLDB may be unavailable). Recovery: supervisor re-checks every 10 seconds; when Orchestrator leader heartbeat resumes, supervisor publishes OrchestratorRecovered to Event Bus and active pipelines resume.
L5 Monitor Health Watch
L5 Monitor publishes a heartbeat event to a dedicated heartbeat:l5 Redis stream every 10 seconds. Supervisor reads this stream independently. On 30-second heartbeat gap: (1) Activates static threshold fallback gate in Orchestrator (CPU>80% → HOLD, error_rate>1% → ROLLBACK). (2) Switches L6 Healing to heuristic-only mode. (3) Fires PagerDuty HIGH. (4) If L5 down >5 minutes and canary is active: supervisor triggers canary rollback directly via kubectl. L5 recovery: supervisor detects resumed heartbeat and publishes L5Recovered → Orchestrator restores full Bayesian gate mode.
RL Model Quality Watch
Every hour, supervisor queries L7 Feedback for the rolling 3-hour RL reward mean and compares to the 7-day baseline. If 3-hour mean drops >15% below 7-day mean: (1) Auto-reverts to previous RL checkpoint in <30 seconds. (2) L6 Healing falls back to heuristic tier only (RL tier disabled). (3) Fires PagerDuty HIGH + schedules L9 retraining. Also monitors KL-divergence: if live RL policy KL-divergence from baseline >0.1 for >2 consecutive hours → same auto-revert path. All revert events written to QLDB with timestamp and trigger reason.
Prompt Quality Watch
Every 6 hours, supervisor samples 50 recent LLM decisions from QLDB and evaluates them against the ground-truth eval set. If accuracy drops >5% from the promoted prompt's original eval score: (1) Flags prompt for L9 review. (2) If accuracy drop >10%: auto-rolls back to previous prompt version. (3) Fires PagerDuty WARN. This catches silent prompt degradation due to LLM model updates or distribution shift — a failure mode not caught by A/B testing alone. Prompt version, accuracy score, and rollback decisions are all written to QLDB.
Event Bus Lag Watch
Supervisor independently checks Redis Streams consumer group lag for all NEEROps consumer groups every 15 seconds. Alert thresholds: lag >5 seconds → PagerDuty WARN; lag >30 seconds → PagerDuty HIGH + Orchestrator pauses new pipeline starts. Dead-letter stream: messages that fail 3 delivery attempts are moved to events:dlq — supervisor alerts on any DLQ message and routes it to human review queue. Supervisor uses a read-only Redis replica connection — not the primary — to avoid adding write load during a potential Redis overload scenario.
Supervisor Self-Monitoring
The supervisor itself is a k8s Deployment with 2 replicas (active-active, since it only reads). k8s liveness probe restarts a failed supervisor pod within 30 seconds. Supervisor writes its own heartbeat to a CloudWatch metric (external to the cluster) every 60 seconds — this enables an external monitoring service (e.g., Datadog, AWS CloudWatch alarm) to alert if the supervisor itself stops running. The external heartbeat check is the one monitoring component that truly has no internal dependency — it uses AWS SDK credentials baked into the supervisor pod's IAM role.
# Autonomy Supervisor — simplified heartbeat check loop
class AutonomySupervisor:
    def run_forever(self):
        while True:
            self.check_orchestrator_heartbeat()    # HTTP, every 5s
            self.check_l5_monitor_heartbeat()      # Redis read-only, every 10s
            self.check_event_bus_lag()             # Redis read-only, every 15s
            self.check_rl_reward_trend()           # QLDB query, every 1h
            self.check_prompt_quality()            # QLDB sample + eval, every 6h
            self.write_own_heartbeat_to_cloudwatch()  # external, every 60s
            time.sleep(5)

    def check_orchestrator_heartbeat(self):
        try:
            resp = requests.get(ORCHESTRATOR_HEALTH_URL, timeout=3)
            if resp.status_code != 200: self.trigger_orchestrator_failover()
        except Timeout:
            self.trigger_orchestrator_failover()  # missed heartbeat → failover

    def trigger_orchestrator_failover(self):
        redis_standby.set("orchestrator:leader", standby_instance_id, nx=True, ex=10)
        if self.no_leader_after(30):  # neither instance recovered
            kubectl.annotate_all_canaries("neerops/pause=true")  # safe pause
            pagerduty.alert(severity="CRITICAL", msg="Orchestrator leader lost — canaries paused")
